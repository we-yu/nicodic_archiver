Summary
-------
Persisted a board-level "Observed Max Res No" per registered target and made it
the user-facing max-response-number column in Registered Articles, replacing the
previously displayed (and misleading) "Saved Max Res No". The observed max is a
monotonic board counter: response content can be deleted/hidden but response
numbers are not removed, so the maximum only ever rises.

Three nullable columns were added to the `target` table via the existing
compatibility-ALTER style. A central monotonic update helper persists observed
values from two sources: the article-top HTML already fetched during
registration (`article_top_preview`) and the scrape-time observed max
(`bbs_page_scrape`). Registered Articles reads the value straight off the target
row (no extra join, no `responses` scan), keeping the MainTask049 latency win.
`saved_max_res_no` remains in `article_response_stats` and on the internal row
dict; it is simply no longer shown in the UI or page CSV.

All 511 tests pass and `./validate.sh` is green. `git diff --check` is clean.

Schema changes
--------------
`target` gains three nullable columns (additive, backward compatible):
- observed_max_res_no INTEGER
- observed_max_res_no_at TEXT
- observed_max_res_no_source TEXT

Added via `_ensure_target_observed_max_res_no_columns()`, called from `init_db()`
alongside the existing `_ensure_target_*` helpers. Existing rows default to NULL.
No index was added (target is small; ordering on the column is already cheap, and
MainTask049 added `idx_target_active_created_at_id`). No change to `responses`,
`articles`, or `article_response_stats` schema.

Storage / update-rule changes
------------------------------
`update_target_observed_max_res_no(...)` behavior:
- observed is None -> no write (reason "observed_none").
- observed not int / bool / negative -> ignored safely, no write.
- stored NULL -> set value + timestamp + source ("advanced").
- observed > stored -> update value + timestamp + source ("advanced").
- observed == stored -> keep value, refresh timestamp (+source)
  ("equal_timestamp_refresh").
- observed < stored -> never lower; timestamp/source untouched ("not_higher").

Implementation uses a guarded UPDATE (`WHERE observed_max_res_no IS NULL OR
observed_max_res_no < ?`) for the monotonic raise, then a second UPDATE keyed on
exact equality for the timestamp refresh — so the raise path is atomic and a
lower value can never win a race. `observed == 0` is honored (empty-board
observation) and only ever sets NULL or refreshes an existing 0.

Parser / extraction changes
----------------------------
`extract_observed_max_res_no(soup)`:
- Reads every `data-res_no` attribute (the same signal `parse_responses` uses)
  and returns the max int when any are present.
- Returns 0 only when a board area is positively rendered (recognized
  `st-bbs*`/`bbs*` markers) but no response numbers are present — an observed
  empty board.
- Returns None for parse miss, unknown/ambiguous shape, or
  unavailable/restricted board (no board area detected).
- Wrapped in a guard so it never raises on unexpected HTML.

Registration-flow changes
--------------------------
`article_resolver._resolve_from_article_url()` already fetches the article-top
soup for identity/title; it now also runs `extract_observed_max_res_no(soup)` on
that same soup (no duplicate HTTP fetch) and returns `observed_max_res_no` in the
success dict. This flows through both URL and exact-title registration paths.
`target_list.register_target_url()` reads it and, when not None, persists it via
the monotonic helper with source `article_top_preview` immediately after
`register_target()`. Failure/parse-miss leaves the target observed max NULL.

Scrape-flow changes
-------------------
`orchestrator.run_scrape()` computes `_observed_max_res_no(json_responses)` and,
after the existing `save_to_db(...)` on the open write connection, calls the
monotonic helper with source `bbs_page_scrape`. Parse-miss/None and lower values
cannot overwrite a higher stored value (helper guarantees).

Hybrid adjustment (adopted from Copilot): the "already up to date" path
(`max_saved_res_no` is not None, `responses` empty, `interrupted` False) now also
persists the saved max as a lower-bound observation. It opens a short write
connection and calls the monotonic helper with source `saved_rows_fallback`
before returning. This lets already-fully-saved targets gradually populate
`target.observed_max_res_no` during periodic random-start scraping, matching the
intended gradual-fill behavior for existing targets. Because the helper is
monotonic, this never lowers a higher existing observed max (e.g. one set earlier
from `article_top_preview`), and it only writes the target row (no response rows,
no `save_to_db`, no change to `article_response_stats`).

Registered Articles UI/CSV changes
----------------------------------
- `REGISTERED_ARTICLE_COLUMNS`: the `saved_max_res_no` entry is replaced (same
  position) by `observed_max_res_no` (label "Observed Max Res No", csv_header
  "observed_max_res_no"). Because the HTML table and page CSV both derive from

- `saved_max_res_no` is still computed and kept on the internal row dict for

Sort / backward-compatibility behavior
---------------------------------------
- `observed_max_res_no` is sortable, added to both the sort allowlist and the
  page-first sort set (it is a pure target column, so it sorts without touching
  stats). ORDER BY places NULLs last in both directions.
- Chosen alias behavior: legacy `sort_by=saved_max_res_no` and
  `sort_by=latest_scraped_max_res_no` now MAP to `observed_max_res_no` (applied
  in both `archive_read.query_registered_articles` and
  `web_app._normalize_registered_sort_by`). Rationale: the user-facing default
  should prefer the observed max, old bookmarked URLs keep working, and the
  saved-stat sort is no longer user-exposed.

Existing runtime DB note
------------------------
Recreating the runtime container does NOT automatically apply the new
compatibility ALTERs to an already-existing `/app/data/nicodic.db`. The new
`target` columns are added by `init_db()` only when it runs against that DB.
A runtime operator should run an explicit `init_db('/app/data/nicodic.db')`
(e.g. via the normal app startup/CLI path that calls init_db) once so the three
observed-max columns are added to the existing target table. Until then,
`archive_read` degrades gracefully: the CTE substitutes NULL for the missing
column and the page renders blank observed values.

Runtime DB safety notes
-----------------------
- All schema changes are additive nullable columns; no data is rewritten,
  renamed, or dropped. `saved_max_res_no` and `saved_response_count` semantics
  are unchanged.
- The monotonic helper never lowers a stored value and never writes NULL over an
  existing value, so a parse miss or a partial/interrupted scrape cannot corrupt

Known limitations
-----------------
- The "already up to date" scrape path now writes a saved_rows_fallback lower
  bound (one bounded target-row UPDATE); the value never regresses.
- Empty-board detection (returning 0) is heuristic on board-area markers; if the
  article-top board preview shape differs from expectations it returns None
  (safe) rather than a wrong number.
- Article-top preview may not always expose response numbers; in that case
  observed max is filled by the next scrape.
- No backfill is performed; existing targets fill in over time.
