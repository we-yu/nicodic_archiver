Auto (Cursor agent router)

Summary
-------
Refactored `query_registered_articles()` to page active registered targets
before reading response statistics for non-aggregate sorts. Default `/registered`
no longer scans/groups all ~13M `responses` rows on every request.

Files changed
-------------
- archive_read.py
- tests/test_archive_read.py

Root cause
----------
Every Registered Articles request joined a global subquery:

  SELECT article_id, article_type, COUNT(id), MAX(res_no)
  FROM responses GROUP BY article_id, article_type

before `LIMIT/OFFSET`, even when ordering by `created_at` or other target fields.

Query strategy before
---------------------
1. Build `resolved_targets` CTE (~12k active targets).
2. LEFT JOIN global `responses` aggregate (all saved rows).
3. ORDER BY + LIMIT/OFFSET.

Query strategy after
--------------------
Shared steps:
- Resolve active targets + article metadata (`resolved_targets` CTE).
- Apply search filter.
- `COUNT(*)` without response stats.

Fast path (`created_at`, `last_scraped_at`, `title`, `article_id`):
1. ORDER BY + LIMIT/OFFSET on target rows only.
2. Fetch `COUNT`/`MAX(res_no)` for page identities via bounded
   `(article_id, article_type) IN (...)` query.
3. Assemble display rows (including zero-response checked `saved_max_res_no=0`).

Aggregate path (`saved_response_count`, `saved_max_res_no`):
1. Join target-scoped stats subquery (responses limited to identities present in
   `resolved_targets`, not entire `responses` table).
2. ORDER BY + LIMIT/OFFSET (or full export when `paginate=False`).

Page-first fast path sorts
--------------------------
- created_at
- last_scraped_at
- title
- article_id

Aggregate ordering sorts
------------------------
- saved_response_count
- saved_max_res_no (including legacy alias `latest_scraped_max_res_no`)

Indexes added
-----------
None.

Future materialized summary compatibility
-----------------------------------------
Page-first path already separates:
- target selection / ordering
- per-identity stats lookup (`_registered_fetch_response_stats`)

A future `articles.saved_response_count`, `saved_max_res_no`, or
`observed_board_max_res_no` can replace the stats helper without changing web
result shape or sort allowlist.

Known limitations
---------------
- Aggregate sorts still require stats for all matching registered articles and
  may remain slow on very large DBs.
- `paginate=False` fast-path export still loads all matching targets then fetches
  stats per identity (~12k), which is acceptable vs 13M-row global aggregate.
- `list_registered_articles()` / scrape_targets.txt export use fast-path sorts
  by default (`created_at`).
