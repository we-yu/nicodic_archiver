Indexes added and rationale
1) idx_articles_type_canonical_url_id on articles(article_type, canonical_url, id)
- Rationale:
  - Supports canonical URL fallback matching by (article_type, canonical_url).
  - Supports stable earliest-row selection by id (MIN(id) lookup path).
  - Targets the strongest bottleneck observed in investigation reports.

2) idx_target_active_created_at_id on target(is_active, created_at, id)
- Rationale:
  - Supports active-target filtering and default Registered ordering workflow.
  - Small-table index only (target table), avoids responses table indexing cost.

Notes:
- No new indexes were added to responses (13M+ rows) to avoid heavy operational cost and because this task focuses on summary-backed reads.

Query-shape changes and rationale
1) Canonical URL fallback path de-correlated
- Before:
  - resolved_targets used a per-target correlated subquery on articles (ORDER BY id LIMIT 1).
- After:
  - Added url_fallback_articles CTE:
    - SELECT article_type, canonical_url, MIN(id)
    - GROUP BY article_type, canonical_url
  - resolved_targets joins this CTE once, then joins articles by selected id.
- Rationale:
  - Removes repeated correlated lookups and allows better use of the new articles index.

2) No-search count fast path
- Before:
  - total count always used resolved_targets CTE count query.
- After:
  - if no search filter, count is computed directly:
    - SELECT COUNT(*) FROM target WHERE is_active = 1
  - search-present count still uses resolved_targets to preserve filtering behavior.
- Rationale:
  - Reduces duplicated expensive work in the common default /registered path.

3) Saved stats source remains summary-first
- _registered_fetch_response_stats remains summary-first and uses bounded responses fallback only for missing summary identities.
- Aggregate-sort branch still joins article_response_stats when table exists.
