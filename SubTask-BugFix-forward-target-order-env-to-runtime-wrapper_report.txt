root cause
- `runtime_local_load` sourced `.env.runtime.local` directly and overwrote already-exported host variables.
- `runtime/periodic_once.sh` forwarded `HOST_CRON_LOG_PATH`, `SOFT_TERMINATE_FILE`, and `ONESHOT_LIMIT_DURATION_SECONDS`, but not `TARGET_ORDER_MODE` or `TARGET_ORDER_START_ARTICLE_ID`.
- Runtime smoke then showed one remaining issue: an empty forwarded `TARGET_ORDER_START_ARTICLE_ID` was treated as an explicit invalid override instead of as “no start override”.

behavior after fix
- `TARGET_ORDER_MODE=reverse ./runtime/periodic_once.sh` now reaches the container process as `TARGET_ORDER_MODE=reverse`.
- `TARGET_ORDER_START_ARTICLE_ID=5400838 ./runtime/periodic_once.sh` now reaches the container process as `TARGET_ORDER_START_ARTICLE_ID=5400838`.
- Empty or whitespace-only `TARGET_ORDER_START_ARTICLE_ID` is now normalized to no override at the target-order config boundary.
- `TARGET_ORDER_MODE=reverse` or `TARGET_ORDER_MODE=random_rotation` now stays effective when start article id is empty or unset.
- `.env.runtime.local` still provides runtime defaults when host inline env is absent.
- Host inline env now overrides `.env.runtime.local` for one invocation.
- Existing `ONESHOT_LIMIT_DURATION_SECONDS`, `SOFT_TERMINATE_FILE`, lock handling, and host cron log flow remain in place.
