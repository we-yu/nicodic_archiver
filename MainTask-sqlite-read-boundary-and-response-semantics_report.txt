SQLite read-boundary changes
----------------------------
Added `storage.open_readonly_db()`:
- Opens existing DB via SQLite URI `mode=ro` (no file creation).
- Returns `None` when the path is missing.
- Sets connect `timeout` and `PRAGMA busy_timeout=5000` (ms).
- Sets `PRAGMA query_only=ON` on file-backed connections.
- Does not run `CREATE TABLE`, compatibility ALTERs, or commits.

Switched read paths from `init_db()` to `open_readonly_db()`:
- archive_read._open_archive_read_conn (all archive read/export/registered queries)
- target_list.list_active_target_urls
- target_list.list_registered_targets
- target_list.inspect_registered_target
- delete_request_feeder.resolve_internal_article_id_input
- delete_request_feeder._load_delete_request_responses
- orchestrator.get_max_saved_res_no
- orchestrator.load_saved_responses
- main.py `export-run-telemetry-csv` (read-only CSV export)

Write paths still use `init_db()` (unchanged):
- target_list.register_target_url, deactivate/reactivate, handoff_redirect
- orchestrator scrape save / queue drain paths
- main._record_scrape_run_observation and other archive writes

`init_db()` now also sets the same busy_timeout on writer connections.

No behavioral change to archive-critical writes (`save_to_db`, target

True observed-board Max Res No
------------------------------
Deferred.

Reason: requires reliable capture at scrape/article-top observation time and a
dedicated persisted field (e.g. on `articles` or telemetry) distinct from saved
`MAX(res_no)`. `orchestrator.ScrapeResult.observed_max_res_no` already exists at
runtime for batch logs but is not persisted as the Registered Articles display
value. Adding schema + backfill in this task would mix historical semantics.
