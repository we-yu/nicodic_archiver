SubTask-zero-response-board-scraped-state — report

Summary
-------
Registered Articles read-side now treats “scrape/check completed but zero saved
responses” as distinct from “never scraped”. Rows use `articles.latest_scraped_at`
(joined as `last_scraped_at`) plus a synthetic Max Res No of `0` when that
timestamp is present and saved response count is `0`. The Web UI highlights
pending rows using the same scrape-completion signal (`last_scraped_at` non-empty
or `saved_response_count` > 0), not saved count alone.

Design decision: representing zero-response scraped state
---------------------------------------------------------
- **Completion signal**: `articles.latest_scraped_at` (already set by `save_to_db`
  on orchestrator scrape completion, including empty `responses` lists).
- **Max Res No**: In `query_registered_articles`, projected column is `0` iff
  `matched_last_scraped_at IS NOT NULL` AND `saved_response_count = 0`; else the
  real `MAX(res_no)` aggregate (possibly NULL only when nothing was ever checked
  according to timestamp rule).
- **UI highlight**: Pending highlight only when there is neither saved responses
  nor a non-empty `last_scraped_at` — backward compatible before
  `latest_scraped_at` existed saved rows still classify as scraped via count.

Known limitations
-----------------
- If `articles.latest_scraped_at` column is absent (unlikely after any
  `storage.init_db` migration) or nothing ever persists that field, behaviour
  remains “pending until first response”; this task does not widen metadata
  beyond existing columns.
- Distinguishes completion via scrape persistence already in product; orchestrator
  was not reworked beyond relying on unchanged `save_to_db` empties-path.
