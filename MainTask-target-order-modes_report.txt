Summary
- Added a bounded target-ordering boundary for batch and periodic scraping.
- Supported default, reverse, random_rotation, and start article override.
- Invalid or unknown configuration falls back to default order without aborting runs.

Implemented modes
- default: preserves loaded target order.
- reverse: reverses the loaded target list.
- random_rotation: picks one start index at run start and rotates once through the list.
- start article override: rotates the loaded target list so the matching article_id is first.

CLI options added
- --target-order-mode MODE
- --target-order-start-article-id ARTICLE_ID

Environment variables
- TARGET_ORDER_MODE
- TARGET_ORDER_START_ARTICLE_ID

Ordering/fallback rules
- CLI options override environment values.
- Unset, empty, or invalid TARGET_ORDER_MODE falls back to default.
- TARGET_ORDER_START_ARTICLE_ID takes precedence over reverse and random_rotation.
- If TARGET_ORDER_START_ARTICLE_ID is empty, invalid, or not found in the active loaded target list, the effective order falls back to default.
- Start override fallback does not continue into reverse or random_rotation.

Precedence rules
- CLI option values
- Environment values
- Default behavior

Article_id override fix
- Active target ordering now reads stored article_id values from the target table for already-registered targets.
- This allows TARGET_ORDER_START_ARTICLE_ID to match numeric article_id even when canonical_url is a slug or encoded /a/<title> URL.
- The final ordered sequence still passed to run_scrape remains canonical URLs.
