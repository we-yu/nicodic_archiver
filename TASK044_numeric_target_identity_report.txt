TASK044 — Numeric NicoNicoPedia target registry identity

Editor AI model
- Composer (Cursor agent); exact SKU not reported by the runtime.

Summary
- Target registration now resolves article pages to a numeric article ID from
  metadata (og:url /id/<digits> with canonical /a/<slug>), stores article_type
  `a`, optional `target.title`, and enforces digits-only `article_id` for type
  `a` at the storage boundary.
- Registered Articles and CSV prefer `target.title`, use stable sort tie-break
  `target.id DESC` after primary sort, and display numeric Article ID for
  digit-stored rows (legacy slug `target.article_id` rows still read for tests).
- Shared parsing lives in `dicopedia_urls.py`; shared metadata extraction in
  `article_page_identity.py` (also used from `orchestrator.fetch_article_metadata`
  numeric path). Imports updated to avoid circular imports.

Schema changes
- `target.title` optional TEXT column via `_ensure_target_title_column` and new
  installs include it. `init_db` applies the migration for existing DBs.

Registration behavior changes
- All intake flows using `register_target_url` / `register_target` resolve via
  `resolve_article_input` (metadata; no live network in unit tests when
  mocked), apply denylist on post-resolve numeric id and URLs, and return
  `resolution_failure` when resolution fails (imports count this key).
- `storage.register_target` validates numeric `article_id` for type `a` (
  `validate_target_article_identity`).
- Redirect handoff re-resolves the redirect canonical URL before inserting the
  new target row.

Registered Articles behavior changes
- `matched_title` prefers non-empty trimmed `target.title`, then saved article
  titles.
- Default sort (`created_at` desc) adds `target_row_id DESC` as tie-breaker.
- Display shows raw digits for numeric `article_id`; non-digit legacy rows still
  use URL-decoded display for compatibility.

Import / feeder / operator CLI behavior changes
- `import_targets_from_text_file` tracks `resolution_failure` per line.
- Delete-request feeder treats `resolution_failure` from registration like other
  non-success outcomes (counts toward failure buckets).
- `main add-target` / `operator_cli` handle the `resolution_failure` outcome for
  registration that cannot complete after URL/metadata resolution.
- Web registration logs `registration_resolution_failure` for metadata
  failures at persistence time after a successful check resolution.

Tests added/updated
- New/updated coverage for resolver ID extraction, denylist id URLs, target
  list mocks, archive read pending/numeric paths, storage validation,
  identity merge and repair slug-only target inserts, and TASK044 integration
  tests.
