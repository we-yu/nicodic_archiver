2. 実装方針
- 保存済み「ニコニコ大百科:掲示板レス削除依頼」(article_id=5511090, type=a)
  のレス本文から、機械的に URL 候補を抽出する。
- URL 候補をカテゴリ判定し、supported なものだけ article target に正規化する。
- 正規化済み候補は裏口登録を作らず、既存の正規登録ルート
  `target_list.register_target_url()` に handoff する。
- 毎回全文走査しないため、`last_processed_res_no` を state として保持し、
  次回はその res_no より後ろのみを処理する。

5. `/id/...` internal helper の bounded 取り込み

    canonical の `article_url` を得られた場合のみ `/a/<slug>` へ解決して handoff
  - ok でなければ skip (推論で補完しない)
  - ユーザ向け article ID 入力 UI/CLI は追加しない

6. last_processed_res_no state の持ち方
- state ファイルは target DB ごとに作成:
  - `<target_db_path>` の suffix を差し替えた
    `*.delete_request_feeder.json`
  - 形式: {"last_processed_res_no": <int>}
- 実行時:
  - `last_processed_res_no` より後ろのみ候補抽出/処理
- 実行後:
  - 保存済みレスの今回観測最大 res_no を `last_processed_res_no` として保存

7. shot 先頭実行と末尾 append の扱い
- `main.run_batch_scrape()` 冒頭で `run_delete_request_feeder()` を一回だけ実行。
- feeder は registry に追加登録を handoff し、その後で既存の
  `list_active_target_urls()` を読み込む。
- registry の insertion order により、新規 target は同一 shot の
  target list の末尾側に自然に出現する。

8. inspect / dry-run seam
- `python main.py delete-request-feeder inspect [--db PATH]`
  - stdout に候補一覧 (res_no/category/raw/normalized/handoff) を出力
  - 末尾に tiny summary を出力
- 通常 batch/periodic 実行時は tiny summary のみ出力
