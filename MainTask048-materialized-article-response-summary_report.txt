Summary
-------
Introduced a bounded materialized per-article saved-response summary table,
`article_response_stats`. Registered Articles now reads saved response stats
from this summary instead of aggregating the full `responses` table on every
request. Aggregate sorts (`saved_response_count`, `saved_max_res_no`) read the
summary directly. Saving responses maintains the summary for the affected
article. An operator-facing rebuild/backfill command (dry-run by default)
recomputes the summary from `responses`.

Schema/table design
-------------------
New table (created by `init_db`, so existing DBs migrate on next writer open):

  CREATE TABLE IF NOT EXISTS article_response_stats (
      article_id TEXT NOT NULL,
      article_type TEXT NOT NULL,
      saved_response_count INTEGER NOT NULL DEFAULT 0,
      saved_max_res_no INTEGER,
      stats_updated_at TEXT NOT NULL DEFAULT (datetime('now')),
      PRIMARY KEY (article_id, article_type)
  )

PRIMARY KEY (article_id, article_type) provides the lookup/upsert index. No
index was added on the 13M-row `responses` table.

Saved response summary semantics
--------------------------------
- `saved_response_count` = COUNT of saved response rows for the identity.
- `saved_max_res_no` = raw MAX(res_no) of saved rows; NULL when no saved rows.
- Zero-response checked articles: stored as `saved_response_count=0`,
  `saved_max_res_no=NULL`. The "checked board with 0 posts shows Saved Max Res
  No = 0" rule remains in the read/display layer, keyed on `last_scraped_at`
  (unchanged from prior behavior). This keeps the stored summary a faithful
  reflection of saved rows and avoids baking display policy into storage.
- Maintenance is recompute-from-responses (bounded to one identity), so
  duplicate `INSERT OR IGNORE` saves never overcount.

Backfill/rebuild behavior
--------------------------
storage.py:
- `recompute_article_response_stats(conn, article_id, article_type)` — single
  identity upsert; called by `save_to_db`.
- `compute_all_article_response_stats(conn)` — full GROUP BY snapshot.
- `rebuild_article_response_stats(conn, *, dry_run=True)` — rebuild whole table.
- `rebuild_article_response_stats_for_db(db_path, *, apply=False)` —
  operator seam; requires explicit existing db_path; dry-run opens read-only
  (no writes), apply opens via init_db and replaces summary rows.
- `format_response_stats_rebuild_lines(db_path, summary)` — operator output.

main.py CLI (mirrors `operator merge canonical-url` maintenance pattern):
  python main.py operator stats rebuild-response-summary --db PATH [--apply]
- Requires explicit --db (refuses implicit runtime DB).
- Dry-run by default; `--apply` performs writes.
- Reports computed articles, existing summary rows, written rows.

Registered Articles query behavior before/after
------------------------------------------------
Before: every request joined a global `responses` GROUP BY
(COUNT(id), MAX(res_no)) before pagination; aggregate sorts ordered on it.

After:
- Fast-path sorts (created_at, last_scraped_at, title, article_id): paginate
  targets first, then read summary stats for the page identities via
  `(article_id, article_type) IN (...)` against `article_response_stats`.
- Aggregate sorts (saved_response_count, saved_max_res_no): LEFT JOIN
  `article_response_stats` directly (no responses aggregation) and ORDER BY the
  summary columns.
- Neither normal-page render nor aggregate sort performs an unbounded full
  `responses GROUP BY` once the summary exists.

Compatibility decisions
------------------------
- Public result shape unchanged; UI labels `Saved Responses` /
  `Saved Max Res No` unchanged.
- Legacy sort alias `latest_scraped_max_res_no` still maps to
  `saved_max_res_no` (unchanged) and is covered by a test.
- Missing summary row (not-yet-backfilled identity) on the fast path falls back
  to a bounded responses aggregation limited to the page identities only.
- Missing summary table entirely (old DB opened read-only before any writer
  init): both paths fall back to the previous responses-based aggregation
  (fast path: page-bounded; aggregate path: target-scoped subquery). No crash.

Known limitations
-----------------
- Aggregate sort correctness at scale depends on the summary being populated.
  Going forward `save_to_db` maintains it; existing data needs a one-time
  backfill (`operator stats rebuild-response-summary --db PATH --apply`).
  Until backfill, aggregate sorts on a large pre-existing DB without the summary
  table fall back to the prior bounded target-scoped aggregation (slower but
  correct); once the table exists but rows are absent, aggregate sort treats
  missing as 0 (COALESCE) — backfill resolves this.
- The rebuild itself performs a full `responses` GROUP BY; it is an explicit,
  operator-controlled maintenance command, not a per-request path.
- True observed-board Max Res No is still not implemented (deferred).
